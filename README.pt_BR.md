## 1. O Problema: BIM de Fachada
- **Cenário**: Projetos públicos onde o IFC é ignorado e apenas PDFs são validados como entrega final.
- **Causa**: A "caneta" (fiscalização/aceite) muitas vezes não possui conhecimento técnico de BIM para validar modelos.
- **Consequência**: O BIM vira apenas um "CAD gourmet". Quem produz modelos ricos vê seu trabalho "apodrecer num HD", enquanto o processo oficial foca apenas na aparência das pranchas 2D.
- **O Agravante da IA**: Em tempos de Inteligência Artificial, dados estruturados são a base de tudo. Validar projetos por PDF hoje não é apenas ineficiência; é tornar o projeto "invisível" para as IAs que poderiam otimizar custos, prazos e manutenção. É rejeitar o maior ganho tecnológico da década.
- **Frase-chave**: *"Se o aceite é PDF, o IFC vira decoração contratual."*

## 2. O Diagnóstico: Aceitação Realista
- **Fato**: A entrega em PDF/DWG é, hoje, inegociável em muitos contextos de mercado e órgãos públicos.
- **Posicionamento**: Não adianta "brigar com o mundo" ou tentar educar o cliente à força no momento da entrega.
- **Histórico**: O BIM (desde as origens com Charles Eastman) sempre visou também facilitar a geração de desenhos e documentação.
- **Conclusão**: A entrega em 2D não invalida o processo BIM. O que invalida o BIM é usar o 2D como fonte de verdade e o modelo como acessório.
- **Frase-chave**: *"PDF não invalida BIM; invalida só o fetiche."*

## 3. Princípios do InfoBIM
1.  **BIM é Processo, não Arquivo**: O valor real está na colaboração, na estruturação de dados e na automação.
2.  **Modelo é Motor, Prancha é Interface**: O modelo trabalha nos bastidores para gerar a documentação necessária.
3.  **Integração**: Quem não sabe BIM é trazido para dentro do fluxo sem precisar "pedir permissão". O processo absorve inputs não-BIM e entrega outputs não-BIM, mas opera em BIM.
4.  **Automação como Cola**: A padronização e a automação garantem que o esforço de modelagem se pague na geração dos entregáveis.

## 4. A Proposta: "BIM por Dentro, 2D por Fora"
Uma metodologia para sobreviver e prosperar no mercado atual, colhendo os benefícios do BIM internamente.

### O Fluxo de Trabalho
1.  **Entrada (Input)**:
    *   Aceita dados não parametrizados ou "sujos" (DWG, PDF, planilhas, levantamentos manuais).
2.  **Núcleo (Core BIM)**:
    *   Onde o trabalho vira dado confiável: Padronização, Estruturação, Classificação, Coordenação e Rastreabilidade.
3.  **Saída (Output)**:
    *   **Obrigatória**: Geração automática de pranchas (PDF/DWG) para cumprir o contrato.
    *   **Opcional**: IFC, BCF, tabelas e dados estruturados para quem tiver maturidade para consumir.

## 5. Como Implementar (Rotina)
- Definir um "contrato interno" do time sobre o que é obrigatório no núcleo BIM.
- Montar o pipeline: templates, folhas, parâmetros mínimos.
- Tratar a prancha sempre como uma **derivação** do modelo, nunca como origem.

## 6. Conclusão
A estratégia não é esperar o mercado amadurecer, mas construir um fluxo onde o BIM vence por "gravidade" e eficiência. Entregamos o que o cliente pede (PDF), mas produzimos da forma que o século XXI permite (Dados).
